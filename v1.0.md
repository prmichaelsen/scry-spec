# Specification: Scry Marker Contract v1.0

**Version**: 1.0.1
**Status**: Active
**Created**: 2026-05-14
**Updated**: 2026-05-15

---

## Overview

This specification defines the **scry marker format** — a structured metadata convention for AI-augmented code and documentation.

**What scry enables**: Scry markers let agents and tools answer questions like:
- *"What code implements this requirement?"*
- *"Which test verifies this scenario?"*
- *"Where is this design decision applied?"*
- *"What's the rationale behind this code?"*

by following lightweight pointers between source code and documentation. A single marker in a source file creates **bidirectional traceability**: from spec to implementation, and from implementation back to spec.

Scry markers are embedded in source files as specially-formatted comments that declare artifact identity, purpose, relationships, and traceability between specifications and implementations.

**Scope of this spec**:
- Marker syntax (sentinels, body format)
- Marker semantics (required/optional fields, validation rules)
- Parsing rules (discovery, extraction, comment stripping)
- Reference format (cross-marker pointers)

**Out of scope**:
- Storage backends (implementations choose: SQL, document store, file scan, RAG, etc.)
- Query interfaces (implementations define: SQL, GraphQL, REST, MCP tools, etc.)
- Indexing strategies (implementations choose: eager, lazy, incremental, etc.)
- Tool surface (implementations define their own CLI/API/MCP surfaces)

**Design Goals**:
- **Human-readable**: Authors can read and edit markers directly
- **Machine-parseable**: Parsers extract structured data reliably
- **Language-agnostic**: Works in any host language with comment syntax
- **Extensible**: Implementations can add fields without breaking compatibility
- **Traceable**: Connect artifacts via line markers
- **Backend-agnostic**: Implementations choose how to store/index/query markers

## Notation

This specification uses the following meta-variables. To extend the spec with new marker types, update the variable definitions — usage sites throughout the spec inherit the change automatically.

### Marker Type Variables

Markers are categorized by **semantic role**, not by syntax (single-line vs multi-line):

| Variable | Expansion | Semantic Role |
|---|---|---|
| `{declarative_marker}` | `entry \| anchor` | Declares a thing to exist (artifact, anchor) with structured metadata |
| `{binding_marker}` | `bind` | Relates one thing to another (source → target) with optional commentary |
| `{marker}` | `{declarative_marker} \| {binding_marker}` | Any scry marker |

**Conceptual model**:
- **Declarative markers** are the "nodes" in scry's knowledge graph — they declare artifacts, anchor points, and similar self-contained entities. They have structured (YAML) bodies with metadata fields.
- **Binding markers** are the "edges" in scry's knowledge graph — they relate one entity to another. They have lightweight positional args (local-id, ref) and optional free-form commentary.

**Syntactic form** is orthogonal to semantic role:
- `{declarative_marker}` markers are always multi-line (delimited by sentinels)
- `{binding_marker}` markers default to single-line, but support a multi-line block form for long commentary (free-form text body, NOT YAML)

**Future markers** should be classified by semantic role:
- A new "declares an X exists" marker goes in `{declarative_marker}` (e.g., `note`, `todo`, `metric`)
- A new "relates X to Y" marker goes in `{binding_marker}` (e.g., `cite`, `alias`, `deprecate`)

### Format Variables

| Variable | Pattern | Description |
|---|---|---|
| `{kind}` | `[a-z]+` | Artifact kind (e.g., `design`, `spec`, `pattern`) |
| `{name}` | `[a-z0-9-]+` | Kebab-case name |
| `{hash}` | `[a-f0-9]{8}` | 8-character hex hash |
| `{id}` | `{kind}.{name}~{hash}` | Full artifact identifier (has `.` before `~`) |
| `{anchor-id}` | `{name}~{hash}` | Anchor identifier (no `.`, just name+hash) |
| `{loose-anchor}` | `[A-Z]+[0-9]+` | Loose anchor name (e.g., `FR1`, `UT3`, `DR2`) |
| `{artifact-ref}` | `{id}[#{loose-anchor}[,{loose-anchor}]*]` | Reference to artifact (with optional loose anchors) |
| `{ref}` | `{artifact-ref} \| {anchor-id}` | Any binding target reference |
| `{local-id}` | `{name}~{hash}` | Local identifier within a file |

**Disambiguation rule**: A `{ref}` is an `{artifact-ref}` if it contains a `.` before the `~`; otherwise it is an `{anchor-id}` (strict anchor reference).

### Extension Guidance

To define a new declarative marker type (declares something to exist):
1. Add to `{declarative_marker}` variable: `entry | anchor | {new_type}`
2. Define its YAML body schema (FRs 4-5 are examples for `entry` and `anchor`)
3. Document its purpose in the Marker Taxonomy section
4. Existing parser/validation rules apply automatically via variable expansion

To define a new binding marker type (relates one thing to another):
1. Add to `{binding_marker}` variable: `bind | {new_type}`
2. Define its inline format (FR2 is the example for `bind`)
3. Document its purpose in the Marker Taxonomy section

## Marker Taxonomy

Scry defines **three marker types** across two semantic categories:

### Declarative Markers `{declarative_marker}`

Markers that **declare something to exist** — artifacts, anchor points, etc. They are "nodes" in the knowledge graph and have structured (YAML) metadata.

| Marker | Purpose |
|---|---|
| `@scry.entry` | Declare an artifact (the primary artifact marker) |
| `@scry.anchor` | Declare a named point within a file |

**Form**: Always multi-line with YAML body between `@scry.{type}` and `@scry.{type}.end` sentinels.

### Binding Markers `{binding_marker}`

Markers that **relate one thing to another** — source-to-target connections. They are "edges" in the knowledge graph and have lightweight references plus optional free-form commentary.

| Marker | Purpose |
|---|---|
| `@scry.bind` | Declare a binding from source to target (universal) |

**Two forms** (semantically equivalent, choose by comment length):
- **Single-line**: `@scry.bind {local-id} {ref} [{comment}]` — most common
- **Block**: `@scry.bind {local-id} {ref}` + multi-line free-form comment + `@scry.bind.end` — for long commentary

Block form contains **free-form text**, NOT YAML (unlike `{declarative_marker}` markers).

### Why Two Categories?

The semantic distinction is **declaration vs relation**:

- A `@scry.entry` declares "this artifact exists" — self-contained, has its own identity, can be referenced
- A `@scry.bind` declares "this thing relates to that thing" — derives meaning from the things it connects

This mirrors common knowledge representation models (RDF, property graphs): things that exist (subjects) vs how they connect (predicates).

### What Are Bindings Actually For?

A binding answers questions like:
- **"What code implements this requirement?"** — Bind code to `spec.X~abcd1234#FR1`
- **"Which test verifies this scenario?"** — Bind test to `spec.X~abcd1234#UT1`
- **"Where is this design decision applied?"** — Bind code to `design.X~def01234#DR2`
- **"Which lesson does this fix address?"** — Bind migration to `lesson.X~cafebabe#L1`
- **"What audit finding does this resolve?"** — Bind code to `audit.X~deadbeef#AF7`
- **"Where is this pattern exemplified?"** — Bind example to `pattern.X~01234567#PT2`

Concrete examples (one line each, showing the typical "this implements that" pattern):

```python
# In auth/jwt.py — JWT validator code
# @scry.bind validate-jwt~a1b2c3d4 spec.auth~abcd1234#FR3
def validate_jwt(token: str) -> bool: ...

# In tests/test_jwt.py — JWT expiry test
# @scry.bind test-expiry~b2c3d4e5 spec.auth~abcd1234#UT1
def test_jwt_expires_after_one_hour(): ...

# In auth/middleware.py — Middleware applies design decision
# @scry.bind auth-middleware~c3d4e5f6 design.architecture~def01234#DR5
class AuthMiddleware: ...

# In migrations/019_fix_csrf.py — Addresses a lesson learned
# @scry.bind csrf-fix~e5f6a7b8 lesson.csrf-bypass~cafebabei2j3k4#L1
def upgrade(): ...

# In components/Counter.tsx — Demonstrates a pattern
# @scry.bind react-counter~d4e5f6a7 pattern.headless-hook~abcd1234#PT2
export const Counter = () => { ... };
```

**Result**: When someone reads `spec.auth~abcd1234#FR3`, they can ask "what implements this?" and scry returns `validate-jwt~a1b2c3d4`. When they read `validate_jwt` in code, they can ask "what does this implement?" and scry returns `spec.auth~abcd1234#FR3`. Bidirectional traceability emerges from a single one-line marker.

#### Why a Single `@scry.bind` Marker?

The binding marker is intentionally generic. It declares: *"this local thing is bound to that target"*. The actual semantic relationship is derived from:

- **Target kind** — `spec.auth~abcd1234` (binding to spec), `pattern.foo~abcdef01` (binding to pattern)
- **Target anchor** — `#FR1` (requirement), `#UT1` (unit test), `#DR2` (design decision)
- **Source context** — Markdown doc, source file, test file
- **Tool interpretation** — Coverage tools, refactor tools, lint tools

This means new binding types (example→pattern, doc→spec, migration→lesson, etc.) require no spec changes. The semantics emerge from the parties involved, not the marker itself.

## Requirements

### FR1: Declarative Marker Sentinel Syntax

**Requirement**: Declarative markers MUST be delimited by opening and closing sentinels.

**Opening sentinel**: `@scry.{declarative_marker}`  
**Closing sentinel**: `@scry.{declarative_marker}.end`

**Per-language forms** — parsers infer comment syntax from YAML body:

**Markdown / HTML**:
```markdown
<!-- @scry.{declarative_marker}
{YAML body}
@scry.{declarative_marker}.end -->
```

**Python / Shell / Ruby / YAML** (`#` comments):
```python
# @scry.{declarative_marker}
# {YAML body}
# @scry.{declarative_marker}.end
```

**TypeScript / Rust / C / Java / JavaScript** (`//` comments):
```typescript
// @scry.{declarative_marker}
// {YAML body}
// @scry.{declarative_marker}.end
```

**SQL / Haskell** (`--` comments):
```sql
-- @scry.{declarative_marker}
-- {YAML body}
-- @scry.{declarative_marker}.end
```

**Lisp / Clojure** (`;` comments):
```lisp
;; @scry.{declarative_marker}
;; {YAML body}
;; @scry.{declarative_marker}.end
```

Parsers MUST strip comment syntax before extracting YAML body.

### FR2: Binding Marker Format

**Requirement**: Binding markers MUST follow one of two forms.

**Single-line form (default)**: `@scry.{binding_marker} {local-id} {ref} [{comment}]`

**Block form (optional, for long comments)**:
```
@scry.{binding_marker} {local-id} {ref}
{multi-line free-form comment}
@scry.{binding_marker}.end
```

The `{comment}` is **optional free-form text** providing context about the binding (status, caveats, cross-refs, etc.).

**Choosing a form**:
- **Single-line**: Default for short comments (one line or less)
- **Block**: Use when commentary exceeds one line — preserves multi-line formatting

**Forms are mutually exclusive**: A binding MUST NOT have both inline comment AND block body. When detected, parsers MUST NOT produce both an inline-comment record and a block-body record from the same opening. Implementations MAY emit diagnostics (warning, info, error) per their strictness mode.

**Block form example**:
```python
# @scry.bind validate-jwt~a1b2c3d4 spec.auth~abcd1234#FR3
# Partial implementation. Currently handles:
#   - JWT signature validation
#   - Token expiry checks
# Pending:
#   - OAuth provider abstraction
#   - Refresh token rotation
# @scry.bind.end
def validate_jwt(): ...
```

**Form disambiguation parsing**:

A line `@scry.{binding_marker} {local-id} {ref} [{trailing-content}]` could represent:
- Single-line binding with no comment (trailing-content absent)
- Single-line binding with comment (trailing-content present)
- Opening of a block-form binding (trailing-content absent, paired with later `.end`)

Parsers MUST disambiguate via **deterministic forward scan**:

1. **Parse the opening line** into `(local-id, ref, trailing-content)` where `trailing-content` may be empty
2. **Scan forward** from the opening line until one of:
   - **`@scry.{binding_marker}.end` is found** → **block form**
     - If `trailing-content` is present → **error** (mixed inline + block, see mutual exclusion above)
     - Otherwise → capture body lines between opening and `.end` as `{comment}`
   - **Another `@scry.{binding_marker}` opening line is found** → previous was **single-line**
     - If `trailing-content` is present → single-line with comment (`trailing-content` becomes `{comment}`)
     - Otherwise → single-line with no comment (`{comment}` is null)
     - Begin disambiguation for the new opening
   - **EOF is reached** → previous was **single-line**
     - If `trailing-content` is present → single-line with comment
     - Otherwise → single-line with no comment

3. **For block form** (when `.end` matches the opening):
   - Strip comment prefix from body lines (per FR11.2)
   - Concatenate body as `{comment}` text, preserving line breaks
   - Unlike `@scry.entry` / `@scry.anchor`, the body is **NOT YAML** — it's free-form text

**Why deterministic scan (no look-ahead window)**:
- Block-form comments can be arbitrarily long (migration notes, audit context, detailed explanations)
- Every implementation produces identical results on identical input
- Matches standard delimiter parsing used in every paired-sentinel system (HTML, XML, code fences, etc.)
- Performance is `O(file_size)` overall — no pathological scan in practice

**Orphaned `.end`**: A `@scry.{binding_marker}.end` line with no preceding unclosed opening MUST NOT produce a marker record. Implementations MAY emit diagnostics (warning, info, error) per their strictness mode — the spec is silent on diagnostic level.

**Block form differences from `@scry.entry`/`@scry.anchor`**:
- No YAML body (free-form text only)
- No structured fields
- Body is purely commentary, not parsed as data

**`{ref}` variants** — two reference modes:

| Mode | Form | Pattern | Meaning |
|---|---|---|---|
| Artifact-level | Loose | `{id}` | Bind to entire artifact (no anchor) |
| Loose anchor (single) | Loose | `{id}#{loose-anchor}` | Bind via convention/structural matching |
| Loose anchor (multiple) | Loose | `{id}#{loose-anchor},{loose-anchor},...` | Bind to multiple loose anchors |
| Strict anchor | Strict | `{anchor-id}` | Bind to explicit `@scry.anchor` marker by ID |

**Examples**:
```python
# === Loose mode (artifact-ref) ===

# Artifact-level binding (no anchor)
# @scry.bind auth-service~a1b2c3d4 spec.auth~abcd1234

# Loose anchor — matches via document structure
# @scry.bind validate-jwt~b2c3d4e5 spec.auth~abcd1234#FR3

# Multiple loose anchors (comma-separated)
# @scry.bind validate-jwt~c3d4e5f6 spec.auth~abcd1234#FR1,FR2,FR3

# === Strict mode (anchor-id) ===

# Strict anchor — references @scry.anchor marker directly by its unique ID
# @scry.bind impl-jwt~e5f6a7b8 token-validation~f1e2d3c4

# === With commentary ===

# Status/caveat
# @scry.bind validate-jwt~a1b2c3d4 spec.auth~abcd1234#FR3 partial impl, OAuth pending

# TODO marker
# @scry.bind login-flow~01234567 spec.auth~abcd1234#FR5 TODO: needs refactor for OAuth

# Disambiguation
# @scry.bind validate~feedface spec.auth~abcd1234#FR3 public API, not internal _validate

# === Cross-kind bindings ===
# @scry.bind jwt-expiry~d4e5f6a7 spec.auth~abcd1234#UT1     # test→spec (loose)
# @scry.bind react-counter~e5f6a7b8 pattern.headless-hook~abcd1234#PT2  # example→pattern (loose)
# @scry.bind auth-explainer~f6a7b8c9 design.architecture~def01234#DR5   # doc→design (loose)
# @scry.bind csrf-check~98765432 csrf-validation~deadbeef   # impl→@scry.anchor (strict)
```

**Components**:
- `{local-id}`: Unique identifier within source file
- `{ref}`: Either `{artifact-ref}` (loose) or `{anchor-id}` (strict)
- `{comment}` (optional): Free-form text providing context about the binding

#### Comment Semantics

The optional `{comment}` is **free-form text**. The spec does NOT define commentary structure.

**Tokenization**: Comments are everything after the 3rd whitespace-separated token. Since `{ref}` contains no whitespace (uses `~`, `.`, `#`, `,` only), splitting is unambiguous:

```
@scry.bind validate-jwt~a1b2c3d4 spec.auth~abcd1234#FR3 partial impl, OAuth pending
└─sentinel┘ └────local-id───────┘ └──────ref──────────┘ └────────comment────────┘
```

**Why comments are first-class**:

Comments are not optional extras — they're how agents and tools discover binding context without reading every source file. The comment field carries the **why** of the binding (status, caveats, intentions, scope) that makes it useful beyond a simple pointer between two IDs.

Without searchable comments, the only way to understand a binding's context is opening the source file. That defeats the purpose of having an indexed knowledge graph.

**Comment requirements**:

- Implementations MUST preserve `{comment}` text verbatim with the binding record
- Implementations exposing query interfaces (MCP tools, SQL, REST, API) MUST include comments in query results
- Implementations providing search MUST make comments searchable (e.g., full-text index, vector embedding, regex)

**Implementation freedom**:

- Search mechanism is implementation-defined (FTS5, Elasticsearch, vector embeddings, regex, etc.)
- Stateless implementations (on-demand file scanners) access comments by re-reading the source — no separate indexing required
- Implementations MAY perform additional analysis (semantic enrichment, named-entity recognition, sentiment, etc.) as a value-added feature
- Tooling presentation (hover tooltips, sidebar lists, decorations) is implementation-defined

#### Disambiguation
- Contains `.` before `~`: artifact-ref (loose mode)
- No `.` before `~`: anchor-id (strict mode)

**See also**: FR14 (Soft References) for how ID-shaped strings appearing in comment text (or anywhere else) are treated — they are NOT formal references and the spec makes no resolution guarantees about them.

**Loose vs Strict — Trade-offs**:

| Aspect | Loose (`#FR1`) | Strict (`anchor-id`) |
|---|---|---|
| Resolution | Convention/structural matching | Direct lookup by ID |
| Robustness | Brittle to renames/restructuring | Stable across document edits |
| Authoring | Just write the convention | Requires explicit `@scry.anchor` |
| Use case | Spec→impl traceability, rapid drafting | Long-term stable references |
| Failure mode | Silent miss if structure changes | Hard error if anchor deleted |

**Expansion semantics for comma-separated loose anchors**:

A single line with N comma-separated loose anchors MUST be expanded to N logical binding records by parsers:

```
# This line:
# @scry.bind validate-jwt~a1b2c3d4 spec.auth~abcd1234#FR1,FR2,FR3

# Expands to 3 binding records:
# 1. (local-id: validate-jwt~a1b2c3d4, ref: spec.auth~abcd1234#FR1)
# 2. (local-id: validate-jwt~a1b2c3d4, ref: spec.auth~abcd1234#FR2)
# 3. (local-id: validate-jwt~a1b2c3d4, ref: spec.auth~abcd1234#FR3)
```

**Note**: Comma-separation is only valid for loose anchors (artifact-ref mode). Strict anchor-ids do NOT support comma-separation — use multiple lines if binding to multiple anchors.

**Range syntax NOT supported in v1.0**:

The spec intentionally does NOT support range syntax like `#FR1..FR5`. Rationale:
- Ambiguous with deletions (`FR1..FR5` after `FR3` removed?)
- Breaks for non-numeric anchors
- Implementations may parse differently
- Comma-separated handles cohesive cases adequately
- True ranges are rare in practice

May be reconsidered in a future spec version if demand emerges.

### FR3: Positional Exclusion Rule

**Requirement**: Binding markers `{binding_marker}` whose offset falls within a declarative marker `{declarative_marker}` span MUST be excluded from indexing.

**Rationale**: Documentation examples often contain marker syntax as text, not as actual markers.

**Example**:
```markdown
<!-- @scry.entry
id: design.example~12345678
kind: design
summary: >
  Here's how to use a bind marker:
  # @scry.bind my-impl~abcd1234 spec.foo~12345678#FR1
  ← This should NOT be indexed
status: active
@scry.entry.end -->
```

The `@scry.bind` inside the doc body is treated as text, not as a real marker.

### FR4: Entry Marker Body (YAML)

**Requirement**: `@scry.entry` markers MUST contain YAML body with these fields.

#### Required Fields (Non-Empty)

These fields MUST be present with non-empty values.

| Field | Type | Description | Validation |
|---|---|---|---|
| `id` | string | Unique identifier | Format: `{id}` |
| `kind` | string | Artifact type | Must be in baseline kind enum (FR8) |
| `summary` | string | One-sentence description | Non-empty |
| `status` | string | Lifecycle state | Must be in status enum (FR9) |

#### Required Fields (Empty Allowed)

These fields MUST be declared in the marker but MAY have empty values. Forcing field presence ensures every marker engages with these dimensions (rationale, applicability, common questions, categorization, priority) and prevents agents from silently ignoring useful indexing data.

| Field | Type | Description | Empty Forms / Default |
|---|---|---|---|
| `rationale` | string | Why this artifact exists | `""`, `null`, `\|` (empty multi-line) |
| `applies` | string | When/where to use this | `""`, `null` |
| `seeded_questions` | array | Common questions answered | `[]`, `null` |
| `tags` | array | Categorization tags | `[]`, `null` |
| `weight` | number | Priority (0.0-1.0) | `null` → defaults to `0.5` |

**Why required-empty instead of optional**: Optional fields tend to be skipped entirely by generators (machines and humans). Required-empty forces the writer to actively consider each field — even leaving it empty is a deliberate signal ("I considered this and have nothing to add") rather than an oversight ("I forgot this field exists").

**`weight` semantics**: When declared with no value (`weight:`, `weight: null`) or omitted from an empty-form template, the value resolves to the default `0.5`. Explicit values override the default. The point of requiring declaration is to force the writer to consider priority — accepting the default is fine, ignoring the field is not.

**Diagnostic guidance**: Implementations MAY emit informational warnings when these fields are empty in kinds where content would typically be expected (e.g., empty `rationale` on a `design` marker is unusual; empty `rationale` on an `internal` marker is normal).

#### Truly Optional Fields

These fields MAY be omitted entirely. They have meaningful "doesn't apply" semantics — most markers genuinely don't have dependencies, supersessions, or implementation links.

| Field | Type | Description | Default |
|---|---|---|---|
| `depends_on` | array | IDs of artifacts this depends on | `[]` |
| `implements` | string | ID of spec/design this implements | `null` |
| `supersedes` | string | ID of artifact this replaces | `null` |

**Example**:
```markdown
<!-- @scry.entry
id: design.auth-flow~a1b2c3d4
kind: design
summary: >
  JWT auth middleware, token validation, refresh flow
status: active
weight: 0.85
tags: ["scope:auth", "topic:security"]
rationale: >
  Missing this causes auth bypass bugs
applies: modifying auth, adding protected endpoints
seeded_questions:
  - How does token refresh work?
@scry.entry.end -->
```

### FR5: Anchor Marker Body

**Requirement**: `@scry.anchor` markers MUST contain a YAML body with these fields.

**Syntax**: `@scry.anchor {local-id}` followed by YAML body and `@scry.anchor.end`.

#### Required Fields (Non-Empty)

| Field | Type | Description | Validation |
|---|---|---|---|
| `name` (in sentinel) | string | Anchor identifier as `{local-id}` (in opening line, not body) | Format: `{name}~{hash}` |
| `description` | string | What this anchor represents | Non-empty |

#### Required Fields (Empty Allowed)

| Field | Type | Description | Empty Forms |
|---|---|---|---|
| `seeded_questions` | array | Common questions about this anchor | `[]`, `null` |

**Why minimal schema**: Anchors are positional markers within parent artifacts. They inherit context (tags, kind, applicability) from the artifact they live in, so they don't need their own categorization fields. The two indexing-useful fields are `description` (what is this anchor for?) and `seeded_questions` (what would someone ask about this location?).

**Why `description` is required-non-empty**: Without a description, an anchor is just `name~hash` — meaningless for browsing, indexing, or navigation. The description plays the same role for anchors as `summary` does for entries.

**Why `seeded_questions` is required-empty**: Same logic as FR4 — optional fields tend to be skipped entirely. Forcing declaration (even if empty) prompts engagement: "are there common questions about this location?" Empty is a deliberate signal, omission is an oversight.

**Example**:
```markdown
<!-- @scry.anchor auth-check~f1e2d3c4
description: JWT validation point for protected routes
seeded_questions:
  - "What triggers JWT re-validation?"
  - "How are revoked tokens detected?"
@scry.anchor.end -->
```

**Minimal valid anchor**:
```markdown
<!-- @scry.anchor auth-check~f1e2d3c4
description: JWT validation point for protected routes
seeded_questions:
@scry.anchor.end -->
```

**Use Cases**:
- Mark specific code locations for cross-reference
- Provide named jump points within large files
- Enable line-marker references to specific code regions

### FR6: Binding Marker References

**Requirement**: Binding markers MUST support both **loose** and **strict** reference modes.

#### Loose Reference Mode (`{artifact-ref}`)

Format: `{id}` or `{id}#{loose-anchor}` or `{id}#{loose-anchor},...`

Resolves via **convention/structural matching** in the target file.

**Examples**:
- `spec.auth~abcd1234` — Reference to spec entry as a whole
- `spec.auth~abcd1234#FR3` — Reference to FR3 (resolved via doc structure)
- `spec.auth~abcd1234#FR1,FR2,FR3` — Reference to multiple loose anchors

#### Strict Reference Mode (`{anchor-id}`)

Format: `{name}~{hash}` (matches an explicit `@scry.anchor` marker's identifier)

Resolves via **direct lookup** of `@scry.anchor` markers.

**Examples**:
- `token-validation~f1e2d3c4` — Strict reference to a specific anchor
- `auth-check~abc12345` — Strict reference to another anchor

**Disambiguation rule**: 
- If `{ref}` contains `.` before `~` → loose mode (artifact-ref)
- If `{ref}` does NOT contain `.` before `~` → strict mode (anchor-id)

#### Valid Targets: What Can Be Bound To

A `@scry.bind` MAY reference:
- An `@scry.entry` marker (whole artifact, via `{id}`)
- A loose anchor within an `@scry.entry` (via `{id}#{loose-anchor}`)
- An `@scry.anchor` marker (strict reference via `{anchor-id}`)

A `@scry.bind` MUST NOT target another `@scry.bind` marker.

**Why**: Bindings are **edges** in the knowledge graph, not nodes. They relate things to other things but do not have first-class identity themselves:

- A binding's `{local-id}` is **file-scoped**, not globally unique — the same `{local-id}` can exist in multiple files independently
- Bindings have no `kind` field, no `{kind}.{name}~{hash}` global ID format
- Allowing edges to target edges would collapse the node/edge distinction, requiring bindings to become hybrid node-edges (identity + relation) — significant spec complexity for marginal benefit
- This mirrors most graph models (property graphs, RDF without reification): edges relate nodes, edges don't get edges

**What happens if you try**:

```python
# File: auth.py
# @scry.bind impl-v1~aabbccdd spec.auth~abcd1234#FR1 v1 implementation

# File: auth_v2.py — trying to bind to the v1 binding
# @scry.bind impl-v2~bbccddee impl-v1~aabbccdd v2 supersedes v1
```

The second binding's `{ref}` (`impl-v1~aabbccdd`) has no `.` before `~`, so per disambiguation it is treated as an `{anchor-id}` (strict mode). The parser looks for an `@scry.anchor impl-v1~aabbccdd` marker — and won't find one (it's a `@scry.bind`, not `@scry.anchor`). The reference is **unresolvable**.

For richer relationship structures than scry models, consider extending the kinds enum (FR8) with a new declarative kind (e.g., `decision`, `assertion`) that has its own identity and can be bound to.

#### Loose Anchor Naming Conventions

For loose references (`#{loose-anchor}`), the spec defines the syntax (`[A-Z]+[0-9]+`) but does NOT enumerate valid prefixes. Anchor semantics are owned by the document being referenced. Common conventions:

| Prefix | Convention Meaning | Typical Target Kind |
|---|---|---|
| `FR<N>` | Functional Requirement | spec |
| `NFR<N>` | Non-Functional Requirement | spec |
| `UT<N>` | Unit Test | spec |
| `DR<N>` | Design Requirement/Decision | design |

**Note**: Documents define their own anchor prefixes. New conventions can emerge without spec changes — the spec only defines the syntax.

#### Resolution: Loose Mode (Structural Matching)

When a reference uses `{id}#{loose-anchor}`, implementations resolve the loose anchor by searching the target file for **where the anchor is defined**, not where it's mentioned. The spec recommends recognizing only patterns that reliably indicate definition (not reference).

**Why definition-only patterns**: Loose anchors are referenced many times throughout a document but defined once. Emphasized text (`**FR1**`, `*FR1*`) is unreliable because authors commonly bold or italicize when *referring to* an FR, not just when *defining* it. Headers, list prefixes, and table cells are conventional definition patterns; emphasis is conventional reference styling.

**Recommended definition patterns** (precedence order, see "First-match wins" below):

**1. Markdown headers** (most reliable):
```markdown
### FR1: User authentication
```
Match pattern: `^#{1,6}\s+{loose-anchor}\b`

**2. List item prefix**:
```markdown
- **FR1**: User authentication
* FR1 — Description
1. **FR1** — Description
```
Match pattern: `^\s*[-*\d.]+\s+\*?\*?{loose-anchor}\b`

**3. First table cell**:
```markdown
| FR1 | User must authenticate |
```
Match pattern: loose anchor appearing in the first cell of a table row (typically a definition column)

**First-match wins**: Within each precedence layer, the **first occurrence** in the file wins. Definitions typically appear before references, so the first match is overwhelmingly the definition. Implementations SHOULD scan from the top of the file and stop at the first match within the highest-precedence layer that contains any match.

**Why not emphasis**: `**FR1**` and `*FR1*` are NOT recommended definition patterns. Many authors bold or italicize anchor names every time they reference them in prose, not only when defining them. Matching on emphasis produces false positives. The spec recommends emphasis NOT be used as a resolution strategy.

**Implementation freedom**:
- Implementations MAY support all three patterns, a subset, or alternative recognition methods
- Implementations SHOULD document which strategies they support
- The precedence order is a **recommendation**, not a requirement
- For non-markdown targets (code, YAML, etc.), implementations define their own loose anchor recognition rules — apply the same principle (recognize definitions, not references)

**Best-effort resolution with artifact fallback**:

Loose anchor resolution is **best effort**. If the anchor cannot be resolved by any supported strategy:
1. The binding is NOT considered failed
2. The binding **falls back to linking the entire artifact** (the `{id}` portion of `{ref}`)
3. The anchor portion (`#{loose-anchor}`) is marked as unresolved but preserved in the record

Example:
```python
# @scry.bind impl-x~aabbccdd spec.auth~abcd1234#FR99
```

If `spec.auth~abcd1234` exists but contains no `FR99` (via any recognition strategy), the binding still links `impl-x~aabbccdd` to `spec.auth~abcd1234` (the artifact). Consumers querying "what implements `spec.auth~abcd1234`?" will see this binding. Consumers querying "what implements `#FR99`?" will not (because resolution failed).

**Why this matters**:
- A typo in the anchor (`#FR99` instead of `#FR9`) doesn't disconnect the binding from the spec entirely
- Bindings remain useful even when the target's anchor structure changes (e.g., FR1 renamed to FR1.a)
- Implementations can surface anchor-resolution warnings for cleanup without breaking bindings

**Diagnostic behavior**: Implementations MAY emit diagnostics (warning, info, error) on anchor resolution failure, per their strictness mode. The artifact-level link MUST still be recorded regardless.

#### Resolution: Strict Mode (Direct Lookup)

When a reference uses `{anchor-id}` (strict mode), implementations MUST resolve by direct lookup:

1. Find `@scry.anchor` marker where the opening sentinel's name matches the `{anchor-id}` exactly
2. If found: binding resolves to that anchor's location
3. If not found: binding is unresolved (hard error or warning, per implementation)

**Strict mode does NOT use structural matching.** A strict reference to `token-validation~abc12345` will only resolve if an `@scry.anchor token-validation~abc12345 ... @scry.anchor.end` exists somewhere in the indexed corpus.

#### Choosing Between Modes

**Use loose mode (`#FR1`) when**:
- Author hasn't placed explicit anchors yet (rapid drafting)
- Document conventions (FR1, DR2, UT3) are the primary structure
- You want bindings to "follow the structure" naturally
- Brittle resolution is acceptable (it's clear when structure breaks)

**Use strict mode (`anchor-id`) when**:
- Long-term stable traceability is critical
- Document may be restructured but anchor identity should remain
- Code-to-spec impl bindings that must not silently drift
- Tools need guaranteed resolution (no structural ambiguity)

### FR7: ID Format

**Requirement**: The `id` field MUST follow the `{id}` format: `{kind}.{name}~{hash}`

**Components**:
- `{kind}`: Artifact type (matches `kind` field for entries)
- `{name}`: Kebab-case name (lowercase, hyphens allowed)
- `{hash}`: 8-character hex hash (collision-resistant identifier)

**Validation regex**: `^[a-z]+\.[a-z0-9-]+~[a-f0-9]{8}$`

**Examples**:
- `design.storage-policy~b43fe7f2`
- `pattern.library-services~a1b2c3d4`
- `spec.auth-flow~f7e8d9c0`

**Hash generation**:
- Implementations MAY use content hash (SHA256 truncated to 8 chars)
- Implementations MAY use random hex string
- Hash MUST be stable across re-generations (prefer content-based)

### FR10: Extension Philosophy (Emergent Standardization)

**Requirement**: Custom kinds MAY be added freely.

#### Philosophy

Custom kinds use the same plain string syntax as baseline kinds:

1. **Use plain kind names**: `kind: tutorial`
2. **Design carefully**: Choose kind names reflecting universal concepts
3. **Accept collision risk**: If scry later adds your kind to baseline, this is usually GOOD
4. **Semantic convergence**: Scry defines baseline kinds based on common usage

#### When Scry Adds Your Kind

**Likely scenario** (zero migration):
- You used `kind: tutorial` for step-by-step guides
- Scry adds `tutorial` to baseline with same semantics
- Your markers work seamlessly

**Rare scenario** (migration needed):
- You used `kind: tutorial` differently than what scry adopts
- You must migrate to a more specific name

#### Design Guidelines

**Universal terms → use carefully**:
- ✅ `tutorial` — if you mean step-by-step guides (universal)
- ❌ `tutorial` — if you mean company-specific onboarding (narrow)

**Narrow terms → safe**:
- ✅ `security-audit-finding` — specific, unlikely to collide
- ✅ `deployment-runbook` — specific enough that collision = alignment

### FR11: Parsing Rules

**Requirement**: Parsers MUST follow these rules.

#### 11.1 Declarative Marker Discovery

**Contract**: A valid declarative marker requires both an opening sentinel (`@scry.{declarative_marker}`) and a matching closing sentinel (`@scry.{declarative_marker}.end`). The body is the text between them.

**On unmatched opening** (no `.end` found): Behavior is implementation-defined. Implementations MAY skip the opening, treat the rest of the file as body until EOF, emit diagnostics at any level, or apply recovery heuristics.

**On unmatched closing** (`.end` with no preceding opening): Behavior is implementation-defined per FR2 (orphan `.end` handling).

#### 11.2 Body Extraction

**Input**: The raw text between the opening sentinel and `.end` sentinel, including any host-language comment prefixes that appear on body lines.

**Output**: Clean YAML text suitable for parsing per FR1's per-language forms.

The mechanism by which an implementation transforms input to output (regex, lexer, AST, heuristics, etc.) is implementation-defined. The spec specifies the contract, not the algorithm.

**Example transformations**:

| Host language | Input | Output |
|---|---|---|
| Markdown (HTML) | `<!-- id: design.X~12345678\nkind: design -->` | `id: design.X~12345678\nkind: design` |
| Python/Shell | `# id: design.X~12345678\n# kind: design` | `id: design.X~12345678\nkind: design` |
| TypeScript/C | `// id: design.X~12345678\n// kind: design` | `id: design.X~12345678\nkind: design` |
| SQL | `-- id: design.X~12345678\n-- kind: design` | `id: design.X~12345678\nkind: design` |

#### 11.3 YAML Parsing
1. Parse YAML body with a YAML 1.2 compliant parser
2. Extract fields into a dictionary or equivalent structure

**On parse failure**: Behavior is implementation-defined. Implementations MAY:
- Skip the marker entirely
- Index whatever fields were successfully extracted (partial indexing)
- Index the marker with a flag indicating parse failure (for surfacing to users)
- Attempt recovery via lenient parsing
- Reject the entire file
- Emit diagnostics at any level (silent, info, warning, error)

The spec does not mandate error-recovery semantics. Different implementations have different tolerance for malformed input — strict tools may reject; lenient tools may salvage. Both are spec-compliant.

#### 11.4 Validation

**Contract**: A marker is **valid** when:
- All required fields are present per its marker type (FR4 for entry, FR5 for anchor, FR2 for binding)
- The `id` field matches the `{id}` pattern (FR7)
- The `kind`, `status`, and other field values are preserved as-authored (FR8, FR9)

**On validation failure**: Behavior is implementation-defined. Implementations MAY skip invalid markers, index them with a validity flag, attempt repair, reject the file, or emit diagnostics at any level. The spec does not mandate failure-handling semantics — strict and lenient implementations are both compliant.

#### 11.5 Unknown Fields
- Parsers MUST NOT error on unknown fields
- Parsers MAY preserve unknown fields in output
- This enables forward compatibility

#### 11.6 Binding Marker Discovery
1. Find a candidate line matching: `@scry.{binding_marker} {local-id} {ref} [{trailing-content}]`
2. Check positional exclusion (FR3) — if inside `{declarative_marker}` span, skip
3. Check inert context exclusion (FR11.7) — if inside an inert region, skip
4. **Disambiguate form** via deterministic forward scan (per FR2):
   - Scan forward from candidate until one of:
     - `@scry.{binding_marker}.end` found → **block form** (validate against `trailing-content` per FR2 mutual exclusion)
     - Another `@scry.{binding_marker}` opening line found → previous was **single-line**; begin disambiguation for the new candidate
     - EOF reached → candidate was **single-line**
5. Extract record: `(local-id, ref, comment)` where:
   - Block form: `comment` is body text between opening and `.end` (prefix-stripped per 11.2)
   - Single-line with comment: `comment` is `trailing-content`
   - Single-line without comment: `comment` is null

**Orphaned `.end`**: A `.end` line not preceded by an unclosed opening MUST NOT produce a marker. Implementations MAY emit diagnostics (warning, info, error) per their strictness mode.

**Note**: The same opening line syntax serves all three forms. The form is determined deterministically by what appears later in the file, not by a length heuristic.

---

#### 11.7 Inert Context Detection

**Requirement**: Parsers MUST NOT extract markers from content that appears inside inert regions. Inert regions are parts of a file that host the *form* of a marker without its *intent* — documentation examples, test fixtures, code string literals, and inline documentation code.

**Rationale**: Spec documentation, test files, and educational prose commonly show marker syntax as text. Without inert-context suppression, parsers mint phantom rows in the index — anchor rows with names like `` ` `` or `,`, doc rows from test fixture content — that pollute the knowledge graph with structurally valid but semantically meaningless records.

**Inert regions MUST include**:

1. **Markdown / text fenced code blocks** — content between a fenced-code opening line and its matching closing line:
   - Opening: a line with 3 or more backtick (`` ` ``) characters at the indent level, optionally followed by a language tag
   - Closing: a line with the same-or-greater count of the same fence character, at the same indent level (CommonMark semantics)
   - The opening and closing fence lines themselves are also inert (the fence character is a structural delimiter, not content)
   - Tilde (`~`) fences follow the same rules as backtick fences
   - A fenced block opened inside another fenced block: the outer fence is authoritative; nested content is inert

2. **Inline code spans** — runs of backtick characters on a single line that delimit inline code:
   - Defined by CommonMark backtick string rules: a run of N backticks opens a span closed by the next run of N backticks on the same line
   - If `@scry.` appears on a line exclusively within such spans (i.e., stripped of all backtick spans, no `@scry.` remains), the entire line is inert for marker purposes

**Inert regions SHOULD include** (implementation-recommended, not required):

3. **Host-language string literals** — content inside string delimiters for the detected host language. The canonical reference behavior is Python triple-quoted strings (both `"""..."""` and `'''...'''`, single-line and multi-line). Implementations that detect the host language from file extension or explicit `language` hint SHOULD suppress markers inside multi-line string literals.

**Inert regions do NOT affect**:
- HTML comments (`<!-- ... -->`) — these are the canonical scry host for markdown files; always parsed
- Line-comment-style blocks (`# ...`, `// ...`, etc.) — these are the canonical scry host for source files; always parsed
- Block-comment-style blocks (`/* ... */`, `/** ... */`, etc.) — always parsed
- Markdown prose outside fenced blocks — always parsed

**Pre-computation requirement**: Implementations MUST compute the full set of inert regions before scanning for markers, not interleaved with marker scanning. This prevents edge cases where a half-parsed marker affects inert-region detection for subsequent lines.

**Example — fenced block (inert)**:
```markdown
The marker format looks like this:

` ` `python
# @scry.entry
# id: design.example~12345678
# kind: design
# summary: Example marker
# status: active
# @scry.entry.end
` ` `

The above block MUST NOT be indexed.
```
(spaces shown in fence delimiters for display; actual fences use unspaced backticks)

**Example — inline code (inert)**:
```markdown
Use `` `@scry.anchor foo~abcd1234` `` in the opening line.
```
The `@scry.anchor` inside the inline code span MUST NOT produce an anchor record.

**Example — real marker after fenced block (NOT inert)**:
```markdown
` ` `python
# @scry.bind phantom~aaaabbbb spec.X~12345678#FR1  ← inert, in fence
` ` `

# @scry.bind real~ccccdddd spec.X~12345678#FR2     ← real, outside fence
```
Only the second binding is indexed.

**Interaction with FR3**: FR3 (positional exclusion of bindings inside declarative spans) and FR11.7 (inert contexts) are orthogonal checks. Both MUST be applied. A line that passes FR3 may still be excluded by FR11.7, and vice versa.

### FR12: Relationship Semantics

**Requirement**: Implementations supporting the `depends_on` field MUST detect cycles.

**Semantics**:
- `depends_on` is a directed edge from marker A to marker B
- Marker A "depends on" marker B (A requires B's existence/correctness)
- Implementations MUST prevent cycles: A → B → C → A

**Cycle detection algorithm** (conceptual):
```
function would_create_cycle(from_id, to_id):
    reachable = traverse_depends_on(to_id)
    return from_id in reachable
```

**Implementation freedom**: How relationships are stored and queried is implementation-defined (SQL tables, graph DB, in-memory dict, on-demand traversal, etc.).

### FR13: Binding Semantics

**Requirement**: `@scry.bind` markers declare a relationship but do NOT define its semantics.

**Semantic resolution** is determined by:

1. **Target kind** — derived from `{ref}` (e.g., `spec.X` = binding to spec)
2. **Target anchor** — derived from `#{anchor}` (e.g., `#FR1` = requirement, `#UT1` = test)
3. **Source context** — file location, language, surrounding code
4. **Tool interpretation** — consumers define what they care about

**Implementations MUST NOT** restrict valid binding combinations. Any source can bind to any target.

**Examples of derived semantics**:

| Binding | Derived Semantic |
|---|---|
| Code file → `spec.X~abcd1234#FR1` | Implementation of requirement |
| Test file → `spec.X~abcd1234#UT1` | Test for unit test case |
| Markdown → `design.X~def01234#DR1` | Documentation of design decision |
| Example → `pattern.X~01234567#PT1` | Example of pattern technique |
| Migration → `lesson.X~cafebabe#L1` | Migration addressing learned lesson |
| Code → `audit.X~deadbeef#AF1` | Code addressing audit finding |

The marker itself is neutral. Tools interpret based on what they need.

### FR14: Soft References

**Requirement**: ID-shaped strings appearing anywhere other than formally specified positions (i.e., the `{ref}` slot of a `@scry.bind` marker) are **soft references**. The spec does NOT treat them as a first-class primitive.

#### Definition

A **soft reference** is any string in any text that matches scry's ID syntax (e.g., `spec.auth~abcd1234`, `pattern.X~cafebabe`, `auth-check~feedface`, `spec.X~abcd1234#FR3`) but is NOT part of a formal scry marker construct.

Soft references can appear anywhere:
- Inside `@scry.bind` comment text
- Inside `@scry.entry` or `@scry.anchor` YAML body fields (e.g., `summary: see spec.X~h for details`)
- In markdown prose outside any marker
- In source-code comments unrelated to scry
- In commit messages, PR descriptions, chat logs, or any other text artifact
- In any file whatsoever

#### Spec-Level Guarantees

The spec defines **only one kind of formal reference**: a `{ref}` field in a binding marker (`@scry.bind`'s 3rd positional slot, per FR6).

Soft references are NOT formal references. The spec guarantees:

- Implementations are NOT required to extract soft references from any location
- Implementations are NOT required to resolve soft references
- Implementations are NOT required to validate soft references (i.e., check that they point to real markers)
- Implementations are NOT required to surface, index, or query soft references
- Soft references have **no defined semantics** at the spec level — they are syntactic coincidence

#### Implementor Freedom

Different implementations have different reasons to care (or not):

- A scry **driver** (indexing engine) typically would NOT care about soft references — its job is to index formal bindings, not interpret narrative text
- A scry **editor extension** (VSCode, JetBrains, Emacs, etc.) MAY want soft references for autocomplete, jump-to-definition, hover previews, or syntax highlighting
- A scry **lint tool** MAY want to validate soft references and warn about orphan IDs
- A scry **documentation tool** MAY want to render soft references as clickable links

What counts as a "valid soft-reference location" is **entirely up to the implementor**. Some examples of choices an extension might make:

- Recognize soft refs only in markdown documentation files
- Recognize soft refs only inside `@scry.bind` comments
- Recognize soft refs anywhere they appear in any file
- Recognize soft refs only in specific YAML fields (e.g., `applies:`, `summary:`)

The spec does NOT guarantee behavior — two compliant implementations might handle soft references completely differently, and that's fine.

#### Why Not Formalize Soft References?

1. **Defining "valid soft-reference location" universally is intractable**: Any text file in a repo could contain ID-shaped strings; deciding which to parse and which to ignore requires arbitrary scope decisions
2. **False positives are costly**: A doc that says "this ID format looks like `kind.name~hash`" would be parsed as containing a soft reference under naive rules
3. **Resolution semantics are unclear**: Should a soft reference to a deleted marker emit a warning? Generate an unresolved entry? Be silently ignored? Different tools want different answers
4. **Pushing this to implementors lets the ecosystem evolve**: An IDE extension can build sophisticated soft-reference UX without spec churn; a lightweight driver can ignore the whole topic

#### Recommended Idiom: Promote Important Soft References to Formal Bindings

If a cross-reference is important enough that you want it tracked, parsed, and queryable, use a formal `@scry.bind` line:

```python
# Not enough — soft reference, no guaranteed tracking
# @scry.bind csrf-fix~f6a7b8c9 audit.security~89abcdef#AF7 also fixes lesson.csrf-bypass~cafebabe

# Better — both relationships are formal bindings
# @scry.bind csrf-fix~f6a7b8c9 audit.security~89abcdef#AF7
# @scry.bind csrf-fix~f6a7b8c9 lesson.csrf-bypass~cafebabe rationale source
def fix_csrf(): ...
```

This makes both relationships explicit and parseable by any spec-compliant implementation, instead of relying on optional soft-reference extensions.

#### Intentional by Design

The emergent behavior is intentional — if something is worth referencing, a marker should be placed. This fulfills the contract of the spec. By explicitly not including soft links in the spec, scry users understand to be intentional about bindings. There is no guarantee a soft link will be honored, which again is intentional and by design. **This is a feature of the spec, not a limitation.**

## Implementation Freedom

**Implementations are free to choose**:

### Storage backends
- **Relational DB** (SQLite, Postgres) — Like reference impl scry-mcp
- **Document store** (MongoDB, CouchDB)
- **Search engine** (Elasticsearch, Meilisearch)
- **Embedded** (LMDB, BadgerDB)
- **Graph database** (Neo4j, NetworkX)
- **Vector store** (Chroma, Qdrant) — for RAG/semantic search
- **In-memory** (for read-only or session-scoped tools)
- **Pure file scan** (no persistent index, query-time discovery)
- **Hybrid** (e.g., index for fast lookup + file scan for verification)

### Query interfaces
- **SQL** — Read-only or read-write queries
- **GraphQL** — Schema-first query API
- **REST** — HTTP-based access
- **MCP tools** — Agent-facing structured access
- **CLI** — Command-line query
- **Library API** — In-process embedded use
- **gRPC/Protobuf** — High-performance RPC

### Indexing strategies
- **Eager** — Index on every change (file watcher)
- **Lazy** — Index on demand (query-triggered)
- **Incremental** — Diff-based updates
- **Batch** — Periodic full scan
- **Hybrid** — Watch + periodic verification

### Tool surfaces
- **MCP server** — Like reference impl scry-mcp
- **Language server** — IDE integration
- **Git hooks** — Pre-commit validation
- **CI integration** — Build-time checks
- **Web UI** — Visual exploration
- **CLI tools** — Scripting and automation

### Validation strictness
- **Strict** — Reject invalid markers, require all fields
- **Lenient** — Best-effort parsing, defaults for missing fields
- **Configurable** — User-defined strictness levels

**The spec is silent on these choices.** Implementations should choose what suits their use case.

## Open Questions

**Q1**: Should `@scry.anchor` support kind field?  
**A1**: Not in v1.0. Anchors are positional markers, not typed artifacts.

**Q2**: Can implementations extend status enum?  
**A2**: Yes, but extended statuses may not interoperate.

**Q3**: What if two markers have the same ID?  
**A3**: Hash collisions should be rare. Implementations MAY detect and warn.

**Q4**: Can I nest markers?  
**A4**: No. Nested markers are not supported in v1.0.

**Q5**: How are kind extensions discovered?  
**A5**: Implementations MAY scan their own data; no central registry.

**Q6**: Must implementations be persistent?  
**A6**: No. Stateless query-time scanners are valid implementations.

**Q7**: Must implementations support all marker types?  
**A7**: Implementations SHOULD support all marker types, but MAY focus on a subset if their use case doesn't require others.

**Q8**: How do I extend `{declarative_marker}` or `{binding_marker}`?  
**A8**: See the Notation section's "Extension Guidance" — update the variable definition and add the new type's body schema. Existing parsing/validation rules apply via variable expansion.

**Q9**: Why no range syntax (`#FR1..FR5`)?  
**A9**: Range syntax has multiple problems: ambiguous with deletions, breaks for non-numeric anchors, implementations may parse differently, and comma-separated form already handles the common case (`#FR1,FR2,FR3,FR4,FR5`). May be reconsidered in future spec versions.

**Q10**: What if a markdown header doesn't match an anchor name exactly?  
**A10**: Per FR6, anchor resolution uses precedence (explicit > header > emphasis > list > table). Implementations SHOULD use word-boundary matching (`\b{anchor}\b`) to handle headers like `### FR1: User auth` matching `#FR1`. If structural recognition fails, fall back to explicit `@scry.anchor` markers.

**Q11**: Can a single bind line reference multiple targets?  
**A11**: No. A bind line has one target `{id}`. The comma-separated form expands anchors WITHIN one target, not multiple targets. To bind to multiple targets, use multiple lines.

**Q12**: When should I use block form vs single-line bind?  
**A12**: Use single-line for short comments (one line or less). Use block form when commentary spans multiple lines — migration notes, detailed status, code review context, etc. Block form preserves line breaks and formatting in the comment body. Both forms produce identical binding records semantically; only the comment text differs.

**Q13**: Why doesn't block-form bind have a YAML body like entry/anchor?  
**A13**: Bindings are relationships, not artifacts. They don't need structured metadata fields (id, kind, status, etc.) — those belong to the target they bind to. The binding itself just needs: local-id, ref, and optional commentary. Block form is purely a syntactic convenience for longer commentary, not a way to add structured fields.

**Q14**: Why "declarative" vs "binding" instead of "block" vs "line"?  
**A14**: "Block" and "line" describe syntax (single-line vs multi-line), but `@scry.bind` can use both forms while remaining semantically a binding. The categories `{declarative_marker}` vs `{binding_marker}` describe **semantic role**:
- **Declarative**: declares something to exist (artifact, anchor) — "nodes" in the knowledge graph
- **Binding**: relates one thing to another (source → target) — "edges" in the knowledge graph

This taxonomy is future-proof. New markers like `@scry.note` (declares a note) or `@scry.cite` (binds code to citation) fit naturally into one category or the other based on their semantic role, regardless of single-line or multi-line form.

## Extension Points

**Implementations MAY**:
- Add custom fields (forward-compatible)
- Recognize additional `kind` values (with sensible defaults)
- Add custom validation rules (stricter than spec)
- Generate markers automatically (editor plugins, CLI tools)
- Provide any tool/query surface
- Choose any storage backend
- Add domain-specific tools (e.g., visualization, lint, refactor)
- Extend `{declarative_marker}` or `{binding_marker}` variables with new types

**Implementations MUST NOT**:
- Change required field semantics
- Require fields not in this spec
- Error on unknown kinds (preserve the original `kind` value as-is per FR8)
- Error on unknown fields (preserve them)
- Reject markers based on storage-specific constraints (storage is impl detail)
- Restrict valid `@scry.bind` source/target combinations (any-to-any is valid)
